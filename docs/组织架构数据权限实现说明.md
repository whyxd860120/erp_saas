# 组织架构与数据权限实现说明

## 一、数据库模型

### 1.1 新增表

#### Department（部门表）
- 支持多级部门树形结构（parentId）
- 关联部门经理（managerId -> Staff）
- 记录部门下职员数量和子部门数量

#### Position（职位表）
- 职位编码、名称、级别
- 支持多个职员担任同一职位
- 记录该职位下职员数量

#### Staff（职员表）
- 职员编码、姓名、手机、邮箱等基本信息
- 关联部门（departmentId -> Department）
- 支持关联系统用户（User）
- 记录管理的客户数量和销售订单数量

#### StaffPosition（职员-职位关联表）
- 支持多对多关系
- 支持主职位标记（isPrimary）

### 1.2 修改的表

#### User（用户表）
- 新增 staffId 字段，关联到 Staff 表
- 一个用户关联一个职员
- 一个职员可以关联多个用户

#### Customer（客户表）
- salesmanId 字段现在关联到 Staff 表
- 支持查看业务员的完整信息

#### SalesOrder（销售订单表）
- salesmanId 字段现在关联到 Staff 表

## 二、数据权限增强

### 2.1 新增值类型

扩展了数据权限的值类型，支持：

1. **current_staff** - 当前用户关联的职员ID
2. **current_dept_staff** - 当前用户部门的所有职员ID列表
3. **current_dept_and_sub_dept_staff** - 当前用户部门及子部门的所有职员ID列表

### 2.2 权限场景示例

#### 场景1：普通业务员
- 配置规则：客户表的 salesmanId = current_staff
- 效果：只能看到自己名下的客户

#### 场景2：部门经理
- 配置规则：客户表的 salesmanId IN current_dept_staff
- 效果：可以看到本部门所有业务员的客户

#### 场景3：大区总监（管理多层部门）
- 配置规则：客户表的 salesmanId IN current_dept_and_sub_dept_staff
- 效果：可以看到本部门及所有子部门业务员的客户

### 2.3 实现原理

1. 通过 `getUserStaffInfo()` 获取用户关联的职员信息
2. 通过 `getDepartmentAndSubDepartmentIds()` 递归获取部门树
3. 根据不同的值类型动态构建过滤条件
4. 支持与现有搜索条件的 AND/OR 组合

## 三、API接口

### 3.1 部门管理
- GET /api/v1/organization/departments - 获取部门列表
- GET /api/v1/organization/departments/tree - 获取部门树
- POST /api/v1/organization/departments - 创建部门
- PUT /api/v1/organization/departments/:id - 更新部门
- DELETE /api/v1/organization/departments/:id - 删除部门

### 3.2 职位管理
- GET /api/v1/organization/positions - 获取职位列表
- POST /api/v1/organization/positions - 创建职位
- PUT /api/v1/organization/positions/:id - 更新职位
- DELETE /api/v1/organization/positions/:id - 删除职位

### 3.3 职员管理
- GET /api/v1/organization/staff - 获取职员列表（支持部门、状态、关键词搜索）
- POST /api/v1/organization/staff - 创建职员
- PUT /api/v1/organization/staff/:id - 更新职员
- DELETE /api/v1/organization/staff/:id - 删除职员

### 3.4 权限控制
- 所有创建、更新、删除操作都需要相应权限
- 部门：department:create/update/delete
- 职位：position:create/update/delete
- 职员：staff:create/update/delete

## 四、前端页面

### 4.1 部门管理（DepartmentListView.vue）
- 树形表格展示部门层级
- 支持新增、编辑、删除部门
- 支持新增下级部门
- 支持设置部门经理
- 显示部门下职员数量和子部门数量

### 4.2 职位管理（PositionListView.vue）
- 列表展示所有职位
- 支持设置职位级别
- 显示该职位下职员数量
- 根据级别显示不同颜色的标签

### 4.3 职员管理（StaffListView.vue）
- 列表展示职员信息
- 支持按部门、状态、关键词搜索
- 显示关联的部门和职位（支持多职位）
- 显示管理的客户数量和订单数量
- 支持关联系统用户
- 显示用户账号状态

### 4.4 用户管理更新（UserListView.vue）
- 新增"关联职员"列
- 新增用户时可选关联职员
- 编辑用户时可以修改关联职员
- 已关联职员的用户不可再选

## 五、使用流程

### 5.1 初始化组织架构
1. 先创建部门（支持树形结构）
2. 创建职位（定义职位级别）
3. 创建职员，关联部门和职位
4. 在职员管理或用户管理中，为职员关联用户账号

### 5.2 配置数据权限
1. 进入"角色权限"页面
2. 选择需要配置的角色
3. 在"数据权限"选项卡中配置规则
4. 例如：业务员角色配置 `客户` 模块，字段 `salesmanId`，操作符 `等于`，值类型 `当前职员`

### 5.3 验证效果
1. 使用配置了权限的用户登录
2. 进入客户管理页面
3. 只能看到自己或本部门业务员的客户

## 六、注意事项

1. **数据权限与搜索条件的组合**
   - 系统会智能合并数据权限条件和用户搜索条件
   - 避免冲突，确保正确的查询结果

2. **删除约束**
   - 删除部门前需要确保没有下级部门和职员
   - 删除职位前需要确保没有职员担任该职位
   - 删除职员前需要确保没有关联的客户和订单

3. **职员与用户的关系**
   - 一个职员可以关联多个用户（但通常是一对一）
   - 用户管理中选择职员时，已关联的职员会被禁用

4. **部门层级**
   - 支持无限层级的部门树
   - 数据权限中 `current_dept_and_sub_dept_staff` 会递归获取所有子部门

## 七、示例数据权限配置

### 7.1 销售业务员角色
```json
{
  "module": "customer",
  "field": "salesmanId",
  "operator": "eq",
  "valueType": "current_staff"
}
```

### 7.2 部门经理角色
```json
{
  "module": "customer",
  "field": "salesmanId",
  "operator": "in",
  "valueType": "current_dept_staff"
}
```

### 7.3 区域总监角色
```json
{
  "module": "customer",
  "field": "salesmanId",
  "operator": "in",
  "valueType": "current_dept_and_sub_dept_staff"
}
```

## 八、扩展性

这个设计可以轻松扩展到其他业务模块：

- 销售订单：按业务员过滤
- 采购订单：按采购员过滤
- 收款单：按业务员过滤
- 付款单：按付款人过滤

只需在对应的数据权限配置中指定相应的字段和值类型即可。
